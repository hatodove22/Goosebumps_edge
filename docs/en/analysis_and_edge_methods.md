# Logging, Analysis, and Edge Inference Implementation Guide (v1)

Updated: 2026-01-16

This document summarizes, for the goosebumps (piloerection) detection project:

- the **logging information required for analysis and training**,
- a **reproducible analysis and training flow** using scripts in `tools/`,
- and two **edge-friendly inference plans** (simple / higher performance) with implementation steps.

The target reader is a first-time implementer who already knows the project goal. The steps and input/output files are made explicit to avoid confusion.

---

## 0. Conclusion (Choose a Plan First)

### Plan A (simple and reliable): Texture index + threshold (session calibration)
- Features: **Laplacian strength (lap_abs_mean / lap_var)** and other light-weight texture indices
- Decision: compute mean/std on baseline (calibration window) and detect with `mean + k*std`
- Add-on: reduce false positives using **quality gates** (IMU, blur, saturation)

Pros:
- No training (minimal GIGO risk)
- Simple to implement, feasible on ESP32-S3
- Works with small labeling effort

Cons:
- Sensitive to individual, lighting, and hair differences (needs tuning)

### Plan B (higher performance): LBP histogram + linear classifier (logistic regression)
- Features: **LBP (Local Binary Pattern) 256-bin histogram**
- Training: `tools/train_lbp_lr.py` (numpy only) and save model as JSON
- Embedding: `tools/export_lbp_lr_header.py` converts to C header for firmware

Pros:
- More robust to variations (lighting, individual) than Plan A
- Low inference cost (LBP + dot product)

Cons:
- Requires labeled data (but still less than deep learning)

## 1. Logging: Required and Recommended Information

The project repeats the loop "data collection -> quality check -> feature check (avoid GIGO) -> training -> embedding".  
Therefore, **logs that can be reproduced later** are critical.

### 1.1 Required (Generated by Collector)
The following must exist under a session directory (e.g., `dataset/subject_001/2026-01-16_session_01/`).

- `meta.json` (fixed conditions)
  - ROI size, lighting settings (PWM range, auto_luma), quality thresholds, FFT settings
- `frames/` + `frames.csv`
  - Frame ID, PC receive time, JPEG path
- `quality.csv`
  - ROI mean luminance, saturation ratios, blur, banding, motion_flag
- `events.csv`
  - Stimulus, goosebumps labels, lighting actions, notes
- (Recommended) `imu.csv`
  - IMU values (g_norm) mapped to frame_id

### 1.2 Required for Training (Generated in post)
- `labels.csv`
  - Teacher labels expanded from `events.csv` goose_on/off
  - Generated by `tools/make_labels.py`

### 1.3 Recommended Events (events.csv)
These are the minimum events needed for the pipeline.

- `calib_start` / `calib_done` (baseline window)
- `stim_on` / `stim_off` (stimulus window)
- `goose_on` / `goose_off` (operator labels)
- `led_pwm` (PWM value in value; note=manual/auto)
- `confound_motion_start/stop` (fixture adjustment etc.)
- `confound_light_start/stop` (light leakage, changes)
- `note` (free text)

## 2. Analysis Flow (When Files Are Needed)

### 2.1 Right After Session Collection
1) `tools/validate_session.py` for basic consistency
2) Run `pilot/report` in UI to check FFT gate (confirm observability and avoid GIGO)

### 2.2 Label Generation
- When `goose_on/off` are present in events.csv, run `tools/make_labels.py` to generate `labels.csv`.

### 2.3 Feature Derivation (Minimum step to avoid GIGO)
- Run `tools/derive_features.py` to compute:
  - FFT texture index (gb_fft_index)
  - Laplacian texture index (lap_abs_mean)
  Then check **AUC and mean gap**.

### 2.4 Training (Plan B)
- Train LBP+LR with `tools/train_lbp_lr.py` (model JSON)
- Validate per session with `tools/infer_lbp_lr.py`
- Export firmware header with `tools/export_lbp_lr_header.py`

## 3. Tools (tools/) and Input/Output

### 3.1 Session Consistency Check
- `tools/validate_session.py <session_dir>`

### 3.2 Label Generation (events -> labels)
- `tools/make_labels.py <session_dir>`

Output:
- `<session_dir>/labels.csv`

### 3.3 Feature Derivation (FFT / Laplacian)
- `tools/derive_features.py <session_dir>`

Outputs:
- `<session_dir>/derived/features.csv`
- `<session_dir>/derived/features_report.json`
- `<session_dir>/derived/features_plot.png` (if matplotlib is available)

Windows (uv) example:
```powershell
cd collector
uv run python ../tools/derive_features.py ../dataset/subject_001/2026-01-16_session_01
```

### 3.4 Training (LBP + Logistic Regression)
- `tools/train_lbp_lr.py --dataset-root <dataset_root> --out <model.json>`

Output:
- `<repo_root>/models/lbp_lr_model.json`

Windows (uv) example:
```powershell
cd collector
uv run python ../tools/train_lbp_lr.py --dataset-root ../dataset --out ../models/lbp_lr_model.json --require-motion-ok
```

### 3.5 Inference (LBP + LR)
- `tools/infer_lbp_lr.py --model <model.json> <session_dir>`

Outputs:
- `<session_dir>/derived/pred_lbp_lr.csv`
- `<session_dir>/derived/pred_lbp_lr_report.json`

### 3.6 Export Firmware Header
- `tools/export_lbp_lr_header.py --model <model.json> --out <header.h>`

Example:
```powershell
cd collector
uv run python ../tools/export_lbp_lr_header.py --model ../models/lbp_lr_model.json --out ../firmware/atoms3r_m12_streamer/include/model_lbp_lr.h
```

## 4. Plan A (Simple and Reliable) Implementation Steps

### 4.1 Goal
- Detect "likely goosebumps" periods without training and use them for:
  - logging
  - validating stimulus efficacy
  - supporting future labeling

### 4.2 Algorithm (Recommended)
1) Get ROI (fixed center ROI)
2) Compute texture index `T` (e.g., `lap_abs_mean`)
3) Compute `mean/std` from calibration window (`calib_start..calib_done`)
4) Threshold `thr = mean + k*std` (k=3 typical)
5) If `T > thr` persists for a duration (e.g., 0.5s), set goose ON
6) Use hysteresis (e.g., OFF at `mean + 2*std`) or moving average to avoid chattering
7) If motion_flag=1 or other quality issues, hold the last state

### 4.3 Implementation Target
- Fastest: **Collector side** (FFT indices already computed)
- Next: **ESP32-S3 side** (Laplacian is light-weight)

### 4.4 Logging (Recommended)
- Append inference ON/OFF to events.csv (e.g., `gb_pred_on/off`)
- Append threshold and calibration stats to meta.json

## 5. Plan B (Higher Performance) Implementation Steps (LBP+LR)

### 5.1 Data Preparation
1) Collect sessions (rig, light blocking, LED control)
2) Record `goose_on/off` in events.csv
3) Generate `labels.csv` with `tools/make_labels.py`
4) Check feature separability with `tools/derive_features.py` (if AUC is too low, redesign)

### 5.2 Training
- Run `tools/train_lbp_lr.py` to create `models/lbp_lr_model.json`
- If possible, validate AUC with session split

### 5.3 Embedding
- Run `tools/export_lbp_lr_header.py` to generate `model_lbp_lr.h`
- Implement in firmware:
  - Resize ROI to `input_size` (e.g., 64)
  - Compute LBP histogram (256 bins)
  - `p = sigmoid(dot(w, hist) + b)`
  - Detect if `p >= threshold`

### 5.4 On-Device Validation
- Send inference results to Collector (add `gb_prob` to HTTP form or send UDP events)
- First check that the inference time series behaves smoothly

## 6. Future Higher-Performance Plan

If moving to deep learning (Tiny CNN), first stabilize:
- ROI size
- Lighting conditions
- Label quality
- Quality gates

Tiny CNN baseline (reference):
- Input: 96x96 grayscale
- Model: a few depthwise separable conv layers
- Quantization: int8
- Runtime: TensorFlow Lite Micro

## 7. Troubleshooting (Analysis and Training)

- Training AUC stuck around 0.5:
  - ROI does not cover skin
  - Light blocking collapsed and illumination changes
  - motion_flag is high (rig is loose)
  - goose_on/off indicates only stimulus presence (not actual goosebumps)

- Inference stays ON:
  - Calibration window includes goosebumps
  - LED is saturated (sat_white_ratio)
